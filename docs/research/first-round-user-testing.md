# SongStyle 首轮用户测试计划

## 研究目标

验证 SongStyle 是否能帮助设计师与 AI 辅助开发者更快地产生清楚、可信、不过度的
Web 设计提案，并识别用户会在哪些地方误解或难以执行这套系统。

本轮测试不预设 SongStyle 有效，也不把视觉偏好等同于设计质量。

## 参与者

- 5–10 位参与者
- 至少包含设计师与使用 AI 编程工具的开发者
- 尽量覆盖不同文化背景与设计经验
- 不要求参与者熟悉宋代艺术或中国设计史

## 测试任务

参与者从同一个产品 Brief 完成两次提案：

1. 使用自己通常的 AI 工作方式，生成普通默认方案。
2. 使用 SongStyle Prompt、原则与审查清单，生成 SongStyle 方案。

任务顺序应在参与者之间交替，减少学习顺序对结果的影响。两个方案必须保留相同的
业务目标、必要内容与可用性要求。

## 推荐 Brief

- 生活方式品牌落地页
- AI 研究工作台落地页
- 数据密集但需要清楚层级的产品页面

## 收集指标

- 形成第一个连贯提案所需时间
- 参与者能否解释保留、删减和重组决策
- 六项共享审查维度得分
- 对层级、信任、可读性与视觉长久性的主观评价
- 参与者是否把 SongStyle 误解为极简主义、古风或低信息密度
- Prompt、Skill、案例与清单中最有帮助和最难执行的部分

## 访谈问题

1. 你认为 SongStyle 解决的核心问题是什么？
2. 哪些信息被保留、重组或删减？为什么？
3. 哪些留白承担了具体功能？
4. 你在哪个时刻认为设计已经足够？
5. 哪些规则妨碍了业务目标或可用性？
6. 你会在真实项目中再次使用哪些资源？

## 证据与产物

每次测试应保留：

- 原始 Brief
- 两个完整方案
- 使用的 Prompt 或 Agent Skill
- 参与者的设计决策说明
- 审查清单评分与访谈笔记

发布社区案例前必须获得参与者许可，并移除敏感或专有信息。

## 分析方法

先比较相同参与者的两个方案，再寻找跨参与者的共同模式。记录 SongStyle 改善、
无影响和造成负面影响的情况。任何结论都必须与实际产物和访谈证据对应。

## 成功与失败信号

积极信号包括更清楚的层级、更少的无功能强调、更容易解释的停止决策，以及未损害
必要信息和业务目标。

失败信号包括页面只是更空、重要内容被删除、所有品牌趋于同一种视觉皮肤，或参与者
无法将原则转化为具体决策。

## 参与方式

请使用 GitHub 的 “SongStyle usage feedback” issue form 提交测试结果，或在提交前通过
“Design discussion” 讨论测试方法。
