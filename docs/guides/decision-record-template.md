# 决策记录模板

SongStyle 的取舍不能只写成“更高级”“更留白”或“更有宋意”。每一次删除、延后、合并、保留和澄清，都要说明它服务了什么任务、牺牲了什么风险、是否可以被复审推翻。

## 使用场景

- 从复杂 Brief 生成页面结构时，记录哪些信息必须保留，哪些可以延后。
- 重写现有页面时，记录每个内容项的变化，避免把必要信息误删。
- 审查设计方案时，把审美判断转成可追溯证据，而不是个人偏好。
- 团队讨论出现分歧时，把争议项拆成用户、业务、可访问性和可读性风险。

## 字段

| 字段 | 填写要求 |
| --- | --- |
| 元素或内容项 | 写清楚被判断的具体模块、文案、控件、图像或交互。 |
| 决策类型 | 只使用删除、延后、合并、保留、澄清五类，避免含混表述。 |
| 理由 | 说明该决策如何服务任务目标、信息秩序或审美表达。 |
| 用户或业务风险 | 记录可能造成的转化、理解、信任、合规或运营损失。 |
| 可访问性或可读性风险 | 记录对比度、字号、焦点状态、标题完整性、辅助技术语义等影响。 |
| 可逆性 | 判断是否容易恢复，并写明恢复条件、成本和责任人。 |
| 复审证据 | 记录数据、用户反馈、对照页面、审查清单或利益相关方确认。 |

## 决策类型

- 删除：内容与当前任务无关、重复或误导，且删除后不破坏用户完成目标。
- 延后：内容重要但不属于当前信息距离，可以进入下级页面、折叠说明或后续流程。
- 合并：多个内容项表达同一意图，可以合成一个更清楚的模块或句组。
- 保留：内容对理解、决策、转化、合规或可访问性必要，不能为了克制而移除。
- 澄清：内容方向正确但表达不完整，需要补充上下文、条件、状态或证据。

## 输出格式

```markdown
## 决策记录：{页面或模块名称}

| 元素或内容项 | 决策类型 | 理由 | 用户或业务风险 | 可访问性或可读性风险 | 可逆性 | 复审证据 |
| --- | --- | --- | --- | --- | --- | --- |
| {具体内容} | {删除/延后/合并/保留/澄清} | {理由} | {风险} | {风险} | {是否可逆及条件} | {证据} |
```

## 复审问题

- 这个决策是否先通过硬约束检查，包括可用性、可访问性、业务目标和必要信息？
- 被删除或延后的内容，是否仍有可发现的路径？
- 被保留的内容，是否有明确任务价值，而不是惯性堆积？
- 被合并的内容，是否仍保持标题、语义和行动路径清楚？
- 可逆性是否真实可执行，而不是事后无法恢复的口头承诺？
