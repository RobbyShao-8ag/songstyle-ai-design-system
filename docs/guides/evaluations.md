# SongStyle 评估方法

固定 brief 用于检查 Prompt 与 Agent Skill 是否在不同任务中保留必要信息、识别风险并给出可执行建议。它们不是为了证明一种审美具有普遍真理。

## 测量什么

- 必须保留的信息是否被保留。
- 已知风险是否被识别或避免。
- 六项共享审查维度是否得到可解释评分。
- 建议是否可以执行。

## 不测量什么

保存结果不声称能够自动判断所有审美价值，也不代替真实用户研究、业务验证或文化研究。

## 如何理解结果

每份结果记录 brief、被测资产、保留项、风险检测、共享清单和 verdict。结果是可审阅证据，不是对未来所有输出的保证。

## 可复现试跑

每次试跑都应保留输入 Brief、AI 输出、审查清单评分和结论，避免只保存主观判断。

1. **输入 Brief**：写清页面类型、目标用户、业务目标、必须保留的信息、已知风险和技术约束。
2. **运行 Prompt 或 Skill**：使用同一份 Brief 分别生成普通 AI 默认方案和 SongStyle 方案，或用 Reviewer Skill 审查同一页面。
3. **保存输出**：把原始输出放入 `evals/results/` 或对应案例目录，记录使用的 Prompt、Skill 版本和日期。
4. **用审查清单评分**：逐项填写信息必要性、层级清晰度、功能性留白、适度停止、秩序而非符号、可用性与目标，并引用证据。
5. **记录失败**：如果必要信息被删除、文化符号替代秩序、留白降低效率，先改进 Prompt 或 Skill，再重新试跑。

优先补充这些案例：

- **SaaS 定价页**：必须保留套餐差异、价格、限制条件、试用路径和购买风险。
- **交易确认页**：必须保留商品或服务、费用、交付条件、风险提示、确认动作和支持路径。
- **数据密集后台**：必须保留指标定义、筛选条件、状态、异常、来源和比较数据。

## 添加新评估

贡献者应在 `evals/briefs.json` 中增加明确 brief、必须保留项与风险，再保存对应结果，并在失败时先改进 Prompt 或 Skill。
